# The Foundry: AI-Powered Structural Engineering Platform

The Foundry is an AI-powered structural engineering platform designed to assist solo engineers by automating Finite Element Analysis (FEA) simulations and providing intelligent design recommendations. It leverages a multi-agent pipeline orchestrated by LangGraph to streamline the analysis and optimization workflow for CAD models.

## Features

The Foundry offers a comprehensive set of features to support structural analysis and design:

*   **CAD Model Upload and Preprocessing:**
    *   Supports upload of `STL`, `STEP`, and `STP` CAD files via a dedicated API endpoint (`/api/upload`).
    *   Automatically converts `STEP` files to `STL` using `gmsh` for consistent mesh processing.
    *   Includes robust mesh repair capabilities utilizing `pymeshfix` to ensure watertight and valid meshes.
    *   Performs surface mesh refinement using `gmsh`, configurable for `low` or `high` resolution.
    *   Tracks and displays mesh triangle counts before and after preprocessing.
    *   Frontend displays estimated part volume, material density, and calculated mass and weight based on user-provided material context.

*   **AI-Powered Force Parsing:**
    *   Utilizes the Google Gemini 2.5 Flash LLM (via `google.generativeai` and a `GEMINI_API_KEY`) to parse natural language descriptions of forces.
    *   Extracts force magnitude, direction, and can map forces to user-defined regions on the model.
    *   Includes a robust fallback parser if the Gemini API is unavailable or fails.

*   **Interactive Force Application:**
    *   The 3D viewer in the React frontend allows users to "paint" specific surface regions on the uploaded model.
    *   Painted regions are used to define precise locations for applying forces and boundary conditions in the simulation.
    *   Brush radius for painting is adjustable.

*   **Finite Element Analysis (FEA) Simulation:**
    *   Performs structural FEA simulations based on the uploaded geometry, defined forces, and material properties.
    *   **High-Fidelity FEniCSx Solver (Optional):**
        *   If `dolfinx`, `mpi4py`, `petsc4py`, `ufl`, and `gmsh` are installed and correctly configured, the platform executes advanced FEA using FEniCSx.
        *   Generates volumetric meshes from STL files using `gmsh`.
        *   Automatically identifies and clamps a fixed support boundary (e.g., the face with the minimum coordinate along the longest dimension).
        *   Applies traction loads based on painted regions and parsed force data.
        *   Solves linear elasticity problems using a Conjugate Gradient (CG) solver with Hypre preconditioners (PETSc options).
        *   Calculates and outputs Von Mises stress distributions.
    *   **Synthetic Fallback Solver:**
        *   If the FEniCSx environment is not fully set up or the real solver encounters an error, a custom Python-only synthetic solver provides approximate stress results.
        *   This solver uses volumetric mesh coordinates (generated by `gmsh` if available, or STL surface vertices as a last resort) and applies mechanics of materials principles.
        *   Includes an automated calibration system to continuously tune its accuracy against analytical benchmark cases (e.g., cantilever beam).
    *   Supports `low` and `high` simulation resolutions.
    *   Outputs maximum/minimum stress, safety factor, and a comprehensive list of stress points for visualization.

*   **Multi-Agent Redesign Pipeline (LangGraph):**
    *   Orchestrates a series of specialized AI agents using a `LangGraph` state machine to automate the simulation, analysis, and redesign process.
    *   **Agents Included:**
        *   **Memory Agent:** Manages session context, stores and retrieves iteration history to an SQLite database (`foundry_memory.db`).
        *   **Geometry Agent:** Analyzes the STL mesh, calculates bounding box dimensions, identifies thin wall regions, sharp interior corners, and through holes, and estimates minimum cross-sectional area and volume. It also identifies geometry-related risk features (e.g., slenderness, thin sections).
        *   **Boundary Condition Agent:** Validates and normalizes user-defined forces and painted regions, ensuring consistent input for the FEA solver.
        *   **Simulation Agent:** Executes the FEA simulation, utilizing either the FEniCSx or synthetic solver.
        *   **Analysis Agent:** Post-processes the simulation results, clusters high-stress zones, calculates local and global safety factors based on the part's material yield strength, and identifies potential failure regions.
        *   **Redesign Agent:** Manages the generation and synthesis of redesign recommendations.
            *   **Parallel Specialist Agents:** Spawns three concurrent `Gemini-2.5-flash`-powered agents using a `ThreadPoolExecutor`:
                *   **Geometry Specialist:** Focuses on shape modifications (fillets, wall thickness, ribs, bosses).
                *   **Material Specialist:** Assesses material selection and suggests alternative alloys or material families.
                *   **Load Path Specialist:** Analyzes force distribution and suggests topology changes for efficient load routing.
            *   **Synthesis Agent:** A `Gemini-2.5-flash`-powered agent that reviews the specialist recommendations, ranks them by predicted impact, identifies potential conflicts, and provides a final, ordered list of design improvements.
            *   Includes robust fallback mechanisms for specialist and synthesis agents if the Gemini API fails or returns invalid responses.

*   **FEA Calibration System:**
    *   An automated calibration process (`backend/utils/calibrate.py`) ensures the synthetic FEA solver provides accurate results.
    *   Generates a standard cantilever beam `STEP` model using `gmsh`.
    *   Compares the synthetic solver's maximum stress output with the analytical solution for the cantilever.
    *   Adjusts a global scaling target (`synthetic_target_pa` in `fea_calibration.json`) to minimize error.
    *   Stores calibration history and state persistently.

*   **Interactive 3D Viewer and UI:**
    *   A responsive React frontend with `react-three-fiber` provides an interactive 3D environment.
    *   Displays uploaded models, force application regions, and stress heatmaps.
    *   UI includes panels for part setup, force annotations, engineer chat, simulation results, and AI redesign recommendations.
    *   Chat interface for system messages, warnings, and agent communication.

## Tech Stack

The Foundry is built with a modern, modular tech stack:

*   **Backend (Python):**
    *   **Web Framework:** Flask
    *   **CORS Handling:** Flask-CORS
    *   **Multi-Agent Orchestration:** LangGraph
    *   **FEA Solver (High-Fidelity):** Dolfinx (with MPI, PETSc, UFL)
    *   **Meshing & Geometry:** Gmsh (Python API binding)
    *   **Mesh Processing:** Trimesh, PyMeshfix
    *   **Numerical Operations:** NumPy
    *   **LLM Integration:** `google-generativeai` (for Gemini 2.5 Flash)
    *   **Data Storage:** SQLite (for session memory)
    *   **Concurrency:** `ThreadPoolExecutor` (for parallel specialist agents)
    *   **Logging:** Custom `SessionLogger`

*   **Frontend (React):**
    *   **Framework:** React
    *   **Build Tool:** Vite
    *   **3D Graphics:** `@react-three/fiber`, `@react-three/drei`, `three.js` (including `STLLoader`)
    *   **Styling:** Custom CSS

*   **External/System Dependencies:**
    *   **CAD/Mesh Engine:** Gmsh (must be installed and accessible in the Python environment)
    *   **FEniCSx Dependencies:** MPI (e.g., Open MPI), PETSc (with Hypre preconditioner) are critical for the high-fidelity FEniCSx solver.

## How It Works

The Foundry streamlines the structural engineering workflow through an integrated frontend and a sophisticated backend multi-agent pipeline:

1.  **Model Upload & Preprocessing:**
    *   An engineer uploads a `STEP` or `STL` CAD file through the React frontend.
    *   The backend receives the file and uses `gmsh` to convert `STEP` to `STL` if necessary.
    *   The mesh is then automatically repaired using `pymeshfix` and refined using `gmsh` to prepare it for simulation.
    *   The processed `STL` is sent back to the frontend for visualization in the 3D viewer.

2.  **Force Definition:**
    *   The engineer describes the forces acting on the part using plain English in the frontend UI.
    *   This natural language prompt is sent to the backend, where the `parse_forces_with_gemini` utility, powered by the Gemini 2.5 Flash LLM, interprets it into structured force data (magnitude, direction).
    *   The engineer then interactively "paints" the precise regions on the 3D model in the frontend where these forces are to be applied.

3.  **Multi-Agent Pipeline Execution:**
    *   Upon initiation of a "Run Simulation" request from the frontend, the backend triggers a `LangGraph` multi-agent pipeline:
        *   **Memory Agent:** Ensures continuity by loading any previous session data or initializing a new session, tracking the iteration number.
        *   **Geometry Agent:** Conducts an in-depth analysis of the part's geometry, identifying critical features (e.g., thin walls, sharp corners) and potential risk factors.
        *   **Boundary Condition Agent:** Validates and normalizes the force inputs, ensuring they are correctly mapped to the painted regions on the model for simulation.
        *   **Simulation Agent:** Executes the core FEA. It first attempts to use the robust FEniCSx solver for high-accuracy results. If FEniCSx is not available or fails, it seamlessly switches to an internally calibrated synthetic solver that provides realistic stress estimations.
        *   **Analysis Agent:** Takes the raw simulation output (stress points) and performs post-processing. It identifies and clusters high-stress zones, calculates safety factors based on the material's yield strength, and determines whether the part passes or fails the design criteria.
        *   **Redesign Agent:** This agent coordinates a "swarm" of three parallel, specialized AI agents (Geometry, Material, Load Path specialists), each powered by Gemini 2.5 Flash. These specialists generate diverse design recommendations. A separate "Synthesis Agent" then evaluates these, ranks them by predicted impact, and resolves any conflicts to produce a concise, actionable list of top recommendations.

4.  **Results & Recommendations:**
    *   The comprehensive simulation results (max stress, safety factor, pass/fail verdict) and the AI-generated redesign recommendations are transmitted back to the frontend.
    *   The frontend visualizes the stress distribution as a color-coded heatmap directly on the 3D model, alongside displaying numerical results and the prioritized list of AI recommendations.

## Local Setup Instructions

This guide assumes you have Python 3.10+, Node.js, npm/yarn, and Git installed on your system.

### 1. Clone the Repository

```bash
git clone <repository-url>
cd the-foundry
```

### 2. Backend Setup

Navigate to the `backend` directory and set up the Python environment.

```bash
cd backend
```

**a. Install Python Dependencies:**

The backend relies on several Python packages. Create a `requirements.txt` file (if not already present) with the following content:

```text
Flask==3.0.3
Flask-Cors==4.0.1
numpy==1.26.4
langgraph==0.0.60
google-generativeai==0.7.1
trimesh==4.3.0
pymeshfix==0.16.1
gmsh==4.12.0
mpi4py==3.1.6
petsc4py==3.21.0
dolfinx==0.8.0
ufl==2024.1.0
```

Then install them:

```bash
pip install -r requirements.txt
```

**b. FEniCSx Installation (Optional, Highly Recommended for Full FEA):**

For the high-fidelity FEniCSx solver to function, `dolfinx` and its dependencies (MPI, PETSc, UFL) must be correctly installed. This is often complex and best managed via Docker or Conda.

**Using Docker (Recommended for FEniCSx):**
The most reliable way to get a working FEniCSx environment is using the official Docker images. You might need to adjust your `backend/app.py` or create a wrapper script to execute FEniCSx simulations within a Docker container if your main backend runs outside it. *Note: The current `backend/core/pipeline.py` expects `dolfinx` to be importable in the Python environment where the Flask app runs.*

For direct installation in your environment (advanced users):
Refer to the official FEniCSx documentation for installation instructions: [https://docs.fenicsproject.org/dolfinx/main/python/installation.html](https://docs.fenicsproject.org/dolfinx/main/python/installation.html)

**c. Set Gemini API Key:**

The AI-powered force parsing and redesign agents require access to the Google Gemini API. Obtain an API key from Google AI Studio and set it as an environment variable:

```bash
export GEMINI_API_KEY="YOUR_GEMINI_API_KEY_HERE"
```

**d. Start the Backend Server:**

```bash
python app.py
```

The Flask server will start, typically on `http://127.0.0.1:5000`.

### 3. Frontend Setup

Open a new terminal, navigate to the `frontend` directory, and set up the React application.

```bash
cd ../frontend
```

**a. Install Node.js Dependencies:**

```bash
npm install
# OR
yarn install
```

**b. Start the Frontend Development Server:**

```bash
npm run dev
# OR
yarn dev
```

The React app will start, usually opening in your browser at `http://localhost:5173`.

### 4. Access the Application

Open your web browser and navigate to `http://localhost:5173`. You should see The Foundry's user interface.

## Agent Architecture

The Foundry's core intelligence is powered by a multi-agent system built with `LangGraph`. Each agent is a distinct node in a state graph, processing and updating a shared `PipelineState`.

**Shared State (`PipelineState`):**
A `TypedDict` that maintains all relevant information across agent interactions, including:
*   `session_id`: Unique identifier for the current user session.
*   `stl_filename`: Name of the active CAD model file.
*   `forces`: List of applied forces with magnitudes, directions, and region IDs.
*   `painted_faces`: Details of user-painted regions (normals, centroids, areas).
*   `resolution`: Simulation resolution (`low` or `high`).
*   `part_context`: User-provided material and part purpose.
*   `geometry_analysis`: Output from the `GeometryAgent`.
*   `boundary_conditions`: Validated force data.
*   `simulation_result`: Raw output from the FEA simulation.
*   `failure_report`: Analysis of simulation results, including safety factors and identified failure zones.
*   `iteration_history`: Historical data for the current session.
*   `iteration_number`: Current simulation-redesign cycle number.
*   `swarm_specialist_outputs`: Raw outputs from individual specialist agents.
*   `redesign_recommendations`: Final, prioritized design recommendations.

**Agent Graph Workflow:**

The agents are connected in a directed graph, ensuring a logical flow through the simulation and redesign process:

1.  **`GeometryAgent` (Entry Point):**
    *   **Input:** `stl_filename`
    *   **Function:** Performs detailed geometric analysis of the STL model, identifying dimensions, features (thin walls, sharp corners, holes), and potential structural risks.
    *   **Output:** Updates `geometry_analysis` in the `PipelineState`.
    *   **Transition:** `GeometryAgent` -> `BoundaryConditionAgent`

2.  **`BoundaryConditionAgent`:**
    *   **Input:** `forces`, `painted_faces`
    *   **Function:** Validates and normalizes the user-defined forces and maps them to the painted regions on the model.
    *   **Output:** Updates `boundary_conditions` in the `PipelineState`.
    *   **Transition:** `BoundaryConditionAgent` -> `SimulationAgent`

3.  **`SimulationAgent`:**
    *   **Input:** `stl_filename`, `boundary_conditions`, `resolution`, `part_context`
    *   **Function:** Executes the FEA simulation using either the FEniCSx or synthetic solver.
    *   **Output:** Updates `simulation_result` in the `PipelineState`.
    *   **Transition:** `SimulationAgent` -> `AnalysisAgent`

4.  **`AnalysisAgent`:**
    *   **Input:** `simulation_result`, `part_context`
    *   **Function:** Interprets the FEA results, identifies high-stress zones, clusters them, calculates safety factors based on the material's yield strength, and generates a failure report.
    *   **Output:** Updates `failure_report` in the `PipelineState`.
    *   **Transition:** `AnalysisAgent` -> `MemoryAgent`

5.  **`MemoryAgent`:**
    *   **Input:** `session_id` (or generates one), `stl_filename`, `forces`, `simulation_result`, `redesign_recommendations` (from previous iteration if any)
    *   **Function:** Manages session history. Loads past iterations for context and prepares to save the current iteration's results.
    *   **Output:** Updates `session_id`, `iteration_history`, `iteration_number` in the `PipelineState`.
    *   **Transition:** `MemoryAgent` -> `RedesignAgent`

6.  **`RedesignAgent`:**
    *   **Input:** `geometry_analysis`, `failure_report`, `iteration_history`, `part_context`, `simulation_result`
    *   **Function:** Orchestrates the generation and synthesis of redesign recommendations:
        *   **Parallel Specialist Agents:** Launches three `Gemini-2.5-flash`-powered agents concurrently (`_geometry_specialist_agent`, `_material_specialist_agent`, `_load_path_specialist_agent`). Each specialist provides a recommendation based on their domain expertise.
        *   **Synthesis Agent:** (`_synthesis_agent`) Consolidates the outputs from the specialist agents, ranks them by predicted impact, identifies conflicts, and produces a final, prioritized list of recommendations.
        *   Saves the complete iteration data (including recommendations) to the `foundry_memory.db` SQLite database.
    *   **Output:** Updates `swarm_specialist_outputs` and `redesign_recommendations` in the `PipelineState`.
    *   **Transition:** `RedesignAgent` -> `END`

This graph-based architecture allows for modularity, clear state management, and the dynamic integration of LLM-powered reasoning with traditional engineering simulation.

## DGX Integration

The provided source code **does not contain any explicit integration or code specific to NVIDIA DGX systems**. The FEniCSx solver is configured to use PETSc's CPU-based Conjugate Gradient (CG) solver with Hypre preconditioners, which are optimized for multi-core CPUs but do not inherently leverage GPUs. While FEniCSx itself can be compiled with GPU support (e.g., using `cuFEM`), this specific implementation does not include the necessary configurations or libraries for such acceleration.

The frontend UI's "DOCKER READY" badge indicates readiness for containerized deployment, which is a common practice for FEniCSx and other HPC applications, but it does not imply DGX-specific hardware utilization or software stack integration within the provided codebase.
